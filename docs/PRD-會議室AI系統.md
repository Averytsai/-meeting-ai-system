## 1. 專案概述

### 1.1 專案背景
企業會議是日常營運的重要環節，但傳統會議記錄方式存在以下痛點：
- 人工記錄耗時且容易遺漏重點
- 會後整理摘要需要額外時間
- 會議資料分散，難以追溯和搜尋
- 外部會議更需要精準的記錄，避免認知落差

### 1.2 解決的問題
| 痛點 | 解決方案 |
|-----|---------|
| 會議記錄耗時 | AI 自動將語音轉為文字並生成摘要 |
| 重點遺漏 | AI 智慧提取重點、決議、待辦事項 |
| 資料分散 | 統一儲存於公司地端資料庫 |
| 會後整理麻煩 | 自動寄送摘要給與會人員 |
| 不知道誰說了什麼 | 說話者辨識功能（進階） |

### 1.3 目標用戶
- **主要用戶**：參與外部會議的業務、PM、主管
- **次要用戶**：需要查閱會議記錄的相關同仁
- **管理者**：IT 人員、系統管理員

## 3. 用戶故事

### 3.1 會議主持人

```
As a 會議主持人
I want 在平板上按下「開始會議」並輸入與會者 Email
So that 系統能自動記錄會議並在結束後發送摘要給所有人
```

**驗收標準：**
- [ ] 可以在平板上看到「開始會議」按鈕
- [ ] 可以輸入多個與會者 Email（支援手動輸入）
- [ ] 開始錄音後有明顯的錄音中提示
- [ ] 可以按下「結束會議」停止錄音

---

### 3.2 與會人員

```
As a 與會人員
I want 在會議結束後收到會議摘要 Email
So that 我可以快速回顧會議重點和待辦事項
```

**驗收標準：**
- [ ] 會議結束後 10 分鐘內收到 Email
- [ ] Email 包含會議基本資訊（日期、時間、與會者）
- [ ] Email 包含 AI 生成的摘要（重點、決議、待辦）
- [ ] 格式清晰易讀

---

### 3.3 系統管理員

```
As a 系統管理員
I want 會議錄音和摘要自動存入公司資料庫
So that 公司可以保留完整的會議記錄供日後查閱
```

**驗收標準：**
- [ ] 錄音檔（音訊格式）正確儲存
- [ ] 會議摘要（文字格式）正確儲存
- [ ] 資料包含必要的 metadata（日期、與會者、會議室等）

│                                         │
│         ┌───────────────┐              │

#### 狀態流程：

```
[待機狀態] ──點擊開始──▶ [錄音中] ──點擊結束──▶ [處理中] ──完成──▶ [待機狀態]
     │                      │                      │
     │                      │                      │
     ▼                      ▼                      ▼
  顯示開始按鈕          顯示錄音時長            顯示處理進度
  可編輯與會者          可新增與會者            「AI 正在生成摘要...」
                       顯示結束按鈕            「正在發送 Email...」
```

### 4.2 AI 摘要格式

```markdown
# 會議摘要

📅 日期：2026-01-05 14:30 - 15:45
📍 地點：會議室 A
👥 與會者：John (john@company.com), Mary (mary@client.com)

---

## 📌 會議重點
1. 討論了 Q1 專案進度，目前進度符合預期
2. 客戶提出新增報表功能的需求
3. 確認下次會議時間為 1/12

## ✅ 決議事項
1. 同意新增月報表功能
2. 預算維持原定金額
3. 專案截止日期延後一週至 2/15

## 📋 待辦事項 (Action Items)
| 項目 | 負責人 | 期限 |
|-----|-------|-----|
| 提供報表規格文件 | Mary | 1/8 |
| 評估開發時程 | John | 1/10 |
| 更新專案時程表 | John | 1/10 |

---
此摘要由 AI 自動生成，如有疏漏請以實際會議內容為準。
```

## 5. 成功指標 (KPI)

### 5.1 MVP 階段目標

| 指標 | 目標值 | 衡量方式 |
|-----|-------|---------|
| 系統可用率 | > 95% | 系統正常運作時間 / 總時間 |
| 語音辨識準確率 | > 85% | 人工抽樣檢查 |
| 摘要生成時間 | < 5 分鐘 | 從會議結束到摘要完成 |
| Email 發送成功率 | > 99% | 成功發送數 / 總發送數 |
| 用戶滿意度 | > 4/5 分 | 使用者問卷調查 |

### 5.2 業務價值指標

| 指標 | 預期效益 |
|-----|---------|
| 節省時間 | 每場會議節省 30-60 分鐘的記錄整理時間 |
| 資訊完整性 | 減少會議重點遺漏情況 |
| 追溯能力 | 100% 會議可追溯查詢 |

## 6. 非功能性需求

### 6.1 效能需求
- 錄音延遲 < 1 秒
- 支援至少 2 小時連續錄音
- 摘要生成時間 < 會議時長的 10%

### 6.2 安全需求
- 錄音檔儲存於地端伺服器，不上傳至公有雲
- 資料庫存取需要驗證
- Email 發送使用公司 SMTP 伺服器

### 6.3 可靠性需求
- 錄音過程中斷電或當機，需保留已錄製內容
- 網路中斷時，本地暫存錄音檔，恢復後再上傳處理

## 8. 風險與假設

### 8.1 假設條件
- 會議室收音設備品質足以支援語音辨識
- 地端伺服器有足夠的運算資源執行 AI 模型
- 與會者願意提供 Email 接收摘要

### 8.2 潛在風險

| 風險 | 影響 | 緩解措施 |
|-----|-----|---------|
| 語音辨識準確率不足 | 摘要品質下降 | 選用高品質收音設備、使用優質 STT 服務 |
| AI 摘要有誤 | 造成誤解 | 加上免責聲明、提供逐字稿供核對 |
| 地端伺服器效能不足 | 處理延遲 | 評估硬體需求、考慮 GPU 加速 |
| 多人同時發言 | 辨識困難 | 建議會議規範、優化收音設備配置 |
